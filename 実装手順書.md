# HMEP レクチャー動画管理パイプライン — 実装手順書

ハワイ大学沖縄事務所 / 2026年5月8日

本書は `要件定義書.md` のフェーズ（F1〜F7）を、そのまま開発作業に落とし込んだ**ステップバイステップの実装手順**です。各ステップの完了条件と確認方法を明記しています。

---

## 0. 作業の進め方（共通ルール）

1. **フェーズ順**で進める（後工程が前工程の成果物に依存するため）。
2. 各フェーズの最後に **「完了条件」を満たしたら** 次へ進む。
3. 認証情報・トークン・個人情報が含まれるファイルは **`credentials/` にのみ配置**し、Git には載せない（`.gitignore` を必ず確認）。
4. 実装中は **`logs/` に日付付きログ**を残す設計を前提にし、動作確認時はログを必ず開いて異常がないか見る。

参照：`要件定義書.md` の「4. ファイル構成」「7. 実装フェーズと目安工数」

## F0. 前提の確認（環境・権限）

**目的：** 手を動かす前にブロッカーを潰す。

| チェック項目 | 確認内容 |
|-------------|----------|
| OS | Windows 10/11 |
| Python | 3.10 以上が `python --version` で確認できる |
| Outlook | Microsoft 365 デスクトップ版が利用可能（`win32com` 用） |
| Google Cloud | プロジェクト作成・YouTube Data API v3 有効化・OAuth クライアント作成が可能なアカウント |
| YouTube | 運用予定チャンネルへのアップロード権限 |

**完了条件：** 上表のいずれかが満たせない場合、要件定義書「6. 制約・前提条件」を満たすまで担当窓口と調整済みであること。

## F1. リポジトリ雛形・仮想環境・設定ファイル

**目的：** `hmep_pipeline/` の骨組みと、再現可能な実行環境を用意する。

### 手順

1. **ディレクトリを作成する**（要件定義書「4. ファイル構成」に準拠）
   ```
   hmep_pipeline/
   ├─ 01_extract/
   ├─ 02_rename/
   ├─ 03_upload/
   ├─ data/
   ├─ videos/
   ├─ videos_renamed/
   ├─ credentials/
   └─ logs/
   ```
2. **仮想環境を作成**し、有効化する（例：`python -m venv .venv` → Activate）。
3. **`requirements.txt` を用意**し、少なくとも以下をバージョン固定で記載する（実装フェーズで不足があれば追記）
   - `pywin32`（Outlook）
   - `openpyxl`（Excel）
   - `PyYAML`（設定）
   - `google-api-python-client`, `google-auth-oauthlib`, `google-auth-httplib2`（YouTube）
4. **`.gitignore` を作成**し、`credentials/`、`token.json`、`.venv/`、`*.xlsx`（運用ポリシーに応じて）、大容量動画、`logs/*.log` などを除外方針で列挙する。
5. **ルートに `config.yaml` の雛形**を置く（フォルダ名・期間・正規表現はダミー可。後続フェーズで中身を確定）。
6. **`README.md`（開発者向け）** に、仮想環境の作り方と「パイプライン実行前に必要なファイル（credentials の配置など）」だけを短く書く（運用マニュアルは F7）。

### 完了条件

- [ ] `hmep_pipeline/` 直下で仮想環境から `python -c "import yaml, openpyxl"` がエラーなく通る
- [ ] `credentials/` が空または `.gitkeep` のみで、**誤コミットされない**構成になっている

## F2. ① メール解析・レクチャー表生成（`01_extract`）

**目的：** Outlook からメールを読み、`lectures.xlsx` を生成する。

### 手順

1. **Outlook 側の前提を固定する**
   - 対象フォルダ名（例：`HMEPレクチャー`）を `config.yaml` に書く。
   - スクリプト実行時は **Outlook が起動済み**であることを README に明記する。
2. **`extract_lectures.py` を実装**
   - `win32com` でフォルダ内メールを走査。
   - 件名・本文から要件定義「3.1.2」のブロックを正規表現で抽出。
   - 抽出結果を行データ化（列は要件定義「3.1.3」のフィールド）。
3. **既存 `lectures.xlsx` との差分・重複防止**ロジックを入れる（同一 `email_subject` や日付+講師など、プロジェクトで採用するキーを決めて実装）。
4. **解析失敗メール**を別シート `failed` 等に出力（要件定義「3.1.3」）。
5. **`data/lectures.xlsx` に出力**（パスは `config.yaml` で変更可能にする）。

### 確認（テスト）

- [ ] 実際またはコピーしたサンプルメールで、**期待列がすべて埋まる行**ができる。
- [ ] 意図的に壊した本文を入れたメールが **`failed` シートに落ちる**。
- [ ] 二回実行しても **重複行が増えない**（または設計通りの挙動）。

### 完了条件

- [ ] `data/lectures.xlsx` が生成され、事務担当が Excel で開いて内容をレビューできる。

## F3. レクチャー表の確認・修正フロー（CLI または Excel）

**目的：** F2 の出力を人が直す運用を、ミスなく回せるようにする。

### 手順（最小構成の例）

1. **レビュー観点を一文でドキュメント化**（日付の typo、講師名の表記ゆれなど）。
2. **CLI の場合：** `lectures.xlsx` を読み、`lecture_date` / `speaker_name` / `title` の一覧表示と、特定行の手動編集を補助するコマンドを用意する。
3. **Excel の場合：** 一覧ビュー用の整形（列固定、フィルタ）をテンプレート化するか、マクロは使わず運用でカバーする方針を決める。

### 完了条件

- [ ] 修正後の `lectures.xlsx` を **F4 の入力としてそのまま使える**。
- [ ] 「誰がいつ確定版にしたか」が分かる運用（ファイル名かコメント列かはチームで決定）が決まっている。

## F4. ② 動画リネーム（`02_rename`）

**目的：** 動画フォルダと `lectures.xlsx` を突合し、`videos_renamed/` にコピー＋リネームする。

### 手順

1. **`config.yaml` に動画ルートパス**（例：`videos/` または `D:\…`）を設定。
2. **`rename_videos.py` を実装**
   - マッチング：要件定義「3.2.1」（更新日時 ±24h + ファイル名類似度）。
   - 命名規則：要件定義「3.2.2」。**禁止文字の除去・置換**を必ず実装。
   - **元ファイルは削除しない**。コピー先は `videos_renamed/`。
3. **リネームログ**を `lectures.xlsx` に追記できる列（元パス、新ファイル名、マッチスコア等）を設計し実装。

### 確認（テスト）

- [ ] 故意に「動画なし」のレクチャー行を入れたとき、**スキップまたはフラグ**が立つ。
- [ ] 同一フォルダに複数候補があるとき、**誤マッチしない**（閾値・確認ログで防ぐ）。

### 完了条件

- [ ] `videos_renamed/` に仕様どおりのファイル名でファイルが並ぶサンプルができる。

## F5. ③ YouTube アップロード（`03_upload`）

**目的：** リネーム済み動画をアップロードし、説明文・限定公開・プレイリスト等を設定する。

### 手順

1. **Google Cloud コンソール作業**
   - YouTube Data API v3 を有効化。
   - OAuth クライアント（デスクトップアプリ等）を作成。
   - `client_secrets.json` を `credentials/` に配置（**Git に含めない**）。
2. **`upload_youtube.py` を実装**
   - 初回ブラウザ認証 → `token.json` を `credentials/` に保存。
   - アップロード前に `upload_log.csv` で **既にアップ済みか確認**（重複防止）。
   - メタデータ：要件定義「3.3.3」（タイトル、説明文、公開設定、プレイリスト、タグ、言語）。
3. **`description_template.txt`** を用意し、要件定義「3.3.4」のプレースホルダと一致させる。
4. **クォータ**（1 本あたりのユニット消費）を意識し、**1 日の処理件数上限**を `config.yaml` で切れるようにする（要件定義「6. 制約・前提条件」）。

### 確認（テスト）

- [ ] **テスト用の短い動画**で限定公開アップロードが成功し、URL が取得できる。
- [ ] 同じファイルを再実行しても **`upload_log.csv` によりスキップ**される。

### 完了条件

- [ ] `upload_log.csv` と `lectures.xlsx` に URL が記録される。

## F6. 統合（`main.py`）および E2E

**目的：** ①→②→③をワンコマンドで回せるようにし、エラー時の再実行も安全にする。

### 手順

1. **`main.py` にサブコマンドまたは引数**を付ける（例：`extract` / `rename` / `upload` / `all`）。
2. **各ステップは単独実行可能**に保つ（要件定義「2.」の「独立して実行可能」）。
3. **ログ：** `logs/pipeline_YYYY-MM-DD.log` にタイムスタンプ付きで集約（要件定義「5.」）。
4. **エラー時の挙動**：要件定義「5. エラーハンドリング方針」に沿って、スキップ・中断・翌日再開の方針をコードに反映。

### 完了条件

- [ ] テスト用データで `all` が通る、または途中失敗から再開できる。
- [ ] ログを見れば「どのファイルがどの理由で失敗したか」が追える。

## F7. 引き渡し（運用マニュアル・セキュリティ）

**目的：** 事務担当が日々運用できる状態にする。

### 手順

1. **運用マニュアル（日本語）**を別紙で作成
   - 実行コマンド、設定の直し方、Outlook/YouTube の事前条件、トラブルシュート。
2. **初回セットアップ手順**（OAuth、フォルダパス、`config.yaml`）を screenshots 付きで整理するか、チェックリストのみにするかを決める。
3. **バックアップ方針**：`lectures.xlsx`、`upload_log.csv`、動画フォルダの退避先。

### 完了条件

- [ ] 担当者以外が読んでも、**週次運用が再現できる**。
- [ ] 認証ファイル・個人情報の取り扱いが文書化されている。
